Slide 1

Discovery

What We Learned Before Changing Code

System Shape

Ship is a unified-document monolith with `web/`, `api/`, and `shared/`, not a set of separate apps.

Core Model

One `documents` table stores docs, issues, projects, and sprints, filtered by `document_type`.

Runtime Flow

The frontend goes through the API for writes, and the editor uses WebSocket + Yjs for realtime collaboration.

Why It Mattered

That discovery told us where improvements had the highest leverage: contracts, first load, hot read paths, and relationship-heavy flows.

Discovery came first so the fixes matched the architecture instead of fighting it.

Category 1

Type Safety

Before

API routes and tests relied too heavily on `!`, `as`, and `any`.

Why It Mattered

TypeScript was present, but the compiler was being bypassed on important auth, input, and response paths.

What Changed

Added checked auth helpers, `zod` validation, honest nullable types, and typed test helpers.

Outcome

Violations dropped from 1292 to 883, a 31.7% reduction.

Slide 3

Category 2

Bundle Size

Before

The app shipped almost the entire frontend on first load, even for lightweight pages.

Why It Mattered

Users were paying for heavy route and editor code before they actually needed it.

What Changed

Added route-level lazy loading, limited devtools to development, and reduced the highlighting payload.

Outcome

Authenticated startup dropped from 2078.55 kB to 485.14 kB, a 76.7% improvement.

Slide 4

Category 3

API Response Time

Before

Protected read endpoints were doing repeated auth and association work on every request.

Why It Mattered

Those fixed costs become bottlenecks as concurrency and data volume grow.

What Changed

Reduced auth overhead, removed an extra issue-association round-trip, and added targeted indexes.

Outcome

P95 improved from 19.00 ms to 14.32 ms on `GET /api/issues`, and from 18.09 ms to 13.58 ms on `GET /api/documents/:id`.

Slide 5

Category 4

Database Query Efficiency

Before

The app was doing unnecessary database work, especially repeated session writes and repeated project-query work.

Why It Mattered

Relationship-heavy flows are where the unified document model gets more expensive.

What Changed

Throttled session `last_activity` writes, tightened the repeated projects query, and added a targeted index.

Outcome

Sprint-board flow dropped from 32 queries to 25, a 21.9% reduction.

Slide 6

Category 5

Test Coverage And Quality

Before

3 critical flows had zero coverage, and web Vitest had 13 failures.

Why It Mattered

The test suite was not strong enough to be a dependable regression safety net.

What Changed

Added tests for CAIA/PIV login start, UI document delete, and sprint-board drag/drop, then stabilized the web suite.

Outcome

Critical uncovered flows dropped from 3 to 0, and web Vitest improved from 138 / 13 to 157 / 0.

Slide 7

Category 6

Runtime Error And Edge Case Handling

Before

Several failure states were silent or confusing, especially around collaboration and title-save failures.

Why It Mattered

Users could miss recovery actions or think failed work had already been saved.

What Changed

Added a global realtime banner, a collaboration disconnect warning, and explicit title-save retry recovery.

Outcome

Console errors during normal usage dropped from 10 to 0, and the three targeted user-facing recovery gaps were fixed.

Slide 8

Category 7

Accessibility Compliance

Before

Repeated low-contrast UI patterns and docs-tree semantics were causing serious accessibility failures.

Why It Mattered

This was both a compliance risk and a usability problem on key pages.

What Changed

Fixed contrast on repeated badges and labels, removed opacity-based contrast loss, and cleaned up docs sidebar tree semantics.

Outcome

Serious automated violations dropped from 3 to 0, contrast failures from 22 to 0, and `/my-week` improved from 96 to 100.

Slide 9

Next Steps

Future Insights

What Still Matters Most

  • Relationship consistency is still the biggest structural pressure point.
  • The realtime layer remains the first likely scale bottleneck.
  • `shared/` should become a more authoritative contract layer.

Recommended Sequence

  • Consolidate relationship handling around the unified document model.
  • Harden scale and operational reliability in the collaboration layer.
  • Finish accessibility validation with full assistive-technology coverage.
  • Keep improving test reliability so the suite remains a trustworthy guardrail.
The strongest path forward is to keep improving the integrated experience, not just individual features in isolation.
